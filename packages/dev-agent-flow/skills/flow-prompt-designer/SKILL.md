# Flow 目标 Agent 提示词设计器（deepagents-flow-ts）

## When to Use

为基于 `deepagents-flow-ts` 开发的**业务 Agent** 设计或重写**系统提示词 / 开场白**时使用。本技能产出的是「被开发的目标 Agent」运行时读取的提示词，不是开发 Agent 自身的提示词。

> 保存与回读统一走 `agent-dev-config`（`4sandbox/agent/dev/config/update` 接口 / `update-prompt` 脚本）。本技能负责「写什么」，`agent-dev-config` 负责「怎么存」。

## 核心原则

1. **结构化 > 自由发挥** —— 模型靠清晰结构和示例稳定输出。提示词必须覆盖下文「七要素」，每个目标 Agent 都按同一骨架填充，不即兴自由写。
2. **few-shot 是质量分水岭** —— 凡输出有固定格式 / 语气 / 结构的场景，**必须**给至少 1 个「输入 → 期望输出」示例。这是「提示词空泛」与「能用」的关键差别。
3. **提示词存平台，不硬编码** —— 经 `agent-dev-config` 保存，运行时统一读取；禁止写进 `src/runtime/` 或节点代码。
4. **保存即回读** —— 每次保存后必须重新 `config` 查询，确认 `systemPrompt` 非空且为新值（治「系统提示词框为空」）。
5. **只改一个字段就省略另一个** —— 单独更新 `systemPrompt` 时**省略** `opening`，反之亦然；传空字符串会覆盖原值（见 `agent-dev-config` SKILL.md:142）。
6. **工具名必须与配置一致** —— 提示词里点名的平台工具，必须是 `agent-dev-config` 已 `tool/add` 进配置的同一个；不要写未配置或臆造的工具名。

## 设计流程（6 步）

| 步 | 动作 | 产出 / 校验 |
|----|------|------------|
| 1 | **场景识别** | 问清：解决什么问题、目标用户、输入输出形态、有无固定输出格式、需要哪些工具/知识库 |
| 2 | **选场景模板** | 从下文「场景模板库」挑最接近的一个作骨架；都不接近则用「通用七要素骨架」 |
| 3 | **填充七要素** | 把模板占位符替换为本场景内容，**至少补 1 个 few-shot** |
| 4 | **质量自检** | 过下文 checklist，逐条满足 |
| 5 | **保存** | `agent-dev-config`：`./scripts/agent_tool.sh update-prompt --text "<提示词>"`（开场白用 `update-opening`） |
| 6 | **回读校验** | `./scripts/agent_tool.sh config \| jq '.data.systemPrompt'` 确认非空且为新值 |

## 七要素骨架（通用，每个系统提示词都应覆盖）

```markdown
# [Agent 名] — [领域] 助手

你是 **[角色]**，服务 **[目标用户]**，专注 **[场景]**。

## 核心能力
- [能力1，动词开头，具体]
- [能力2]
- [能力3]
（3–5 条，避免「无所不能」式空话）

## 工作方式
按工作流图运行：理解需求 → 按需调用工具 → 观察结果 → 给出回答。
- 信息足够直接答；不足先用工具查或向用户提问，不臆测。
- 工具失败时读错误、换参数或换工具，不原地打转。

## 工具使用
- [工具A]：什么场景用、关键入参（工具名须与平台配置一致）
- [工具B]：...
（没有外部工具就写「本 Agent 不依赖外部工具，直接用 LLM 回答」）

## 领域知识与约束
- [规则 / 术语 / 边界 / 禁区]

## 输出规范
- 格式：[结构 / 字段 / Markdown]
- 语气：[正式 / 亲和 / 专业]
- 长度：[简洁 / 详细]

### 示例（few-shot，按需 1–2 个）
输入：[典型输入]
输出：[符合上述规范的期望输出]

## 兜底
- 信息不足：列出还需要什么，向用户提问。
- 超出范围：说明边界，给可行替代或引导。
```

## 场景模板库（挑最近的一个改）

### A. 智能客服 + 知识库（RAG 问答）
特征：基于知识库答用户问、多轮、需引用来源、不知道就承认。
要点：
- 能力强调「先检索知识库再回答、引用出处、拒绝编造」。
- 工具：知识检索工具（平台 Knowledge，经 agent-dev-config 配置）。
- few-shot 必给：含「检索到 → 引用作答」与「检索不到 → 礼貌兜底」各一例。
- 输出规范：先结论后依据；附来源；未命中时给出转人工/补充信息路径。

### B. 内容生成（如小红书爆款笔记）
特征：按主题产出风格化文案、有固定结构（标题/正文/标签/emoji）。
要点：
- 能力强调「套用爆款结构、控制字数、风格一致」。
- few-shot 必给：1 个完整「主题 → 成品笔记」示例，体现标题钩子、分段、话题标签。
- 输出规范：明确字段（标题≤20字 / 正文分段 / 3–6 个 #标签 / emoji 适度）。
- 约束：不堆砌违禁词、不虚假宣传。

### C. 数据分析助手（数据表 / SQL / 看板）
特征：对接数据表或 SQL API、按维度聚合、给结论+解读。
要点：
- 能力强调「先确认口径与时间范围、再查询、最后解读」。
- 工具：数据表 / SQL 查询（平台 Plugin/Workflow，经 agent-dev-config 配置）；SQL 契约写入 `project.md`。
- few-shot 必给：1 个「自然语言问题 → 查询参数 → 结论解读」示例。
- 输出规范：结论先行 + 关键数字 + 简短归因；必要时给表格。
- 约束：口径不明先问，不臆造字段；只读不写。

### D. 任务工具型 Agent（调用 API / 执行动作）
特征：把用户意图翻成工具调用、串接多步、回报结果。
要点：
- 能力强调「解析意图 → 选对工具 → 校验入参 → 执行 → 回报」。
- 工具：相应 Plugin/Workflow（经 agent-dev-config 配置，入参以平台 schema 为准）。
- few-shot 必给：1 个「请求 → 工具调用 → 结果回报」示例。
- 输出规范：动作结果清晰可读；失败给原因与下一步。
- 约束：副作用操作（写/删/付费）执行前向用户确认。

## 质量自检 checklist（保存前逐条过）

- [ ] 开头一句话说清「你是谁 + 服务谁 + 做什么」
- [ ] 核心能力 3–5 条，动词开头、具体，无「全能」空话
- [ ] 工具指引存在，且工具名与平台配置一致（无臆造/未配置工具）
- [ ] 有领域约束 / 边界 / 禁区
- [ ] 输出规范明确（格式 / 语气 / 长度）
- [ ] **至少 1 个 few-shot**（凡输出有固定格式必给）
- [ ] 有兜底（信息不足 / 超范围怎么办）
- [ ] 无自相矛盾的指令、无过时工具名
- [ ] 已保存到平台并 `config` 回读确认非空为新值

## Anti-patterns

- ❌ 提示词只有「角色 + 几句空泛能力」，无工具指引、无 few-shot、无输出规范（典型「空泛」）。
- ❌ 列「全能」能力（「能处理一切问题」），模型反而失焦。
- ❌ 提示词里写未在 `agent-dev-config` 配置的工具名，或与配置 `targetId` 对不上。
- ❌ 输出有固定格式却不给示例，靠模型自由发挥。
- ❌ 把提示词硬编码进 `src/runtime/` 或节点代码，而非存平台。
- ❌ 单独改 `systemPrompt` 时把 `opening` 传成空串，覆盖原开场白。
- ❌ 保存后不 `config` 回读就声称「已配置好」。
- ✅ 按七要素骨架 + 场景模板填充，必带 few-shot，存平台后回读校验。
